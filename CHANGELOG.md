# Changelog

All notable changes to this project are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project will adhere to [Semantic Versioning](https://semver.org/spec/v2.0.0.html) once a numbered release is cut.

## [Unreleased]

### Added
- **`feat(a5)`** Per-device scrcpy stream settings modal on all device dashboards. "Stream Settings" in quick-actions opens a settings modal (video codec, encoder, bitrate, max FPS, max resolution, audio toggle, audio source, audio codec) with probe-driven smart defaults from ws-scrcpy-web's device probe. Settings persist in the database and are passed to ws-scrcpy-web as URL params on the embed iframe. First visit probes the device automatically with a 3-second grace period before connecting — single clean connection, no double-connect. Three modal buttons: Save (persists without reconnecting, stays open), Connect (applies settings and reconnects, closes modal), Clear/Refresh (re-probes device for updated capabilities, stays open). Two-column grid layout matching ws-scrcpy-web's ConfigureScrcpy modal.
- **Android Power Tools module** (new, sort order 2, sits between Android Devices and Jellyfin in the sidebar). Hosts ws-scrcpy-web's full home page via iframe at `/android-power-tools` — one-click shell (xterm), file browser (ListFilesModal), stream configuration, network scan + manually-add-device panel, and dependency updater. Peer of the Android Devices module, not a replacement — the existing device management surface (registered-devices list, PIN unlock, power state, sleep/wake, screensaver, Projectivy backups) stays untouched. Module declares no dependencies of its own (inherits ws-scrcpy-web from AndroidDevices). Fallback: if `WsScrcpy.IsRunning` is false, shows a warning alert instead of a broken iframe.
- `CHANGELOG.md` (this file) following Keep a Changelog format.
- `LICENSE` — GPL-3.0-only (matches ws-scrcpy-web, the primary external dependency).
- `SECURITY.md` — private vulnerability reporting flow via GitHub Security Advisories, 72-hour acknowledgement SLA.
- `CONTRIBUTING.md` — setup, build/test commands, project structure, module architecture, code style, PR guidelines.
- `.gitignore` entries for `.claude/` (Claude Code session artifacts), `src/ControlMenu/dependencies/go2rtc/*` (runtime-managed binaries, matching existing pattern for other dep folders), and `src/ControlMenu/go2rtc.yaml` (runtime-generated config rebuilt on startup from the cameras table).
- `DeviceType` enum gained `AndroidTablet` and `AndroidWatch` alongside the existing `GoogleTV` / `AndroidPhone`. Persisted as string via EF so existing rows are untouched; no migration needed. Tablet and watch devices don't yet get a dedicated dashboard, but they're first-class in the Device Management table and the mDNS scan auto-classifier.
- `ScrcpyMirror.razor` gained an optional `DeviceKind` parameter (`"phone"`, `"tablet"`, or `"tv"`). When set, it's forwarded to ws-scrcpy-web's `/embed.html` as `&deviceKind=<value>` so the stream toolbar's D-pad/Touch input-mode toggle seeds the right default. `GoogleTvDashboard.razor` passes `"tv"` (D-pad default); `PixelDashboard.razor` passes `"phone"` (Touch default). Requires ws-scrcpy-web commit `6f217cd` or later, which wires the URL param through in `embed-entry.ts`.
- `IAdbService.ScanMdnsAsync()` returns a list of `MdnsAdbDevice` records (service name + IP + advertised ADB port) by parsing `adb mdns services`. Uses a direct port of ws-scrcpy-web's tab-split `parseMdnsOutput` for robustness.
- `IAdbService.DetectDeviceKindAsync(ip, port)` classifies an Android device as `"phone"`, `"tablet"`, `"tv"`, or `"watch"` using five parallel shell probes (`ro.build.characteristics`, `pm has-feature android.software.leanback`, `pm has-feature android.hardware.type.watch`, `wm size`, `wm density`). Mirrors ws-scrcpy-web's four-signal `classifyDeviceKind` plus a fifth watch-feature probe for Wear OS support. Watch precedence > TV > tablet/phone smallestWidthDp split.
- `IAdbService.GetPropAsync(ip, port, property)` — thin `adb shell getprop <name>` wrapper returning the trimmed value (or empty string on failure). Used by the Discovered panel to pre-fill a newly-added device's Name from `ro.product.model` when no name is remembered.
- **"Discovered on Network" panel** in Settings > Devices. After a Scan, mDNS-advertised devices that aren't yet registered show up in a dedicated table with IP, advertised ADB port, MAC (resolved via ARP — a ping pass refreshes ARP for any mDNS IPs not already in the table), and a one-click Add button. Clicking Add opens the Device form pre-filled with MAC, IP, port, and an initial Type of `AndroidPhone`; a background ADB probe then refines Type (`DetectDeviceKindAsync`) and auto-fills Name from `ro.product.model` when the name wasn't already restored from the delete-remembers-name flow.
- **Delete remembers the device's name.** When a device is deleted from Settings > Devices, its user-assigned name is stashed in the Settings table at `device-name-<mac>`. When the same MAC later reappears via mDNS scan and the user clicks Add, the form pre-fills Name from that stashed value before falling back to the ADB product-model probe — the user doesn't retype "My Phone" just because they deleted and re-added the device.
- **Network scanner modal** — `Settings › Devices` gained a new *Scan Network…* button that opens a full scanner modal (CIDR/IP/range subnet input, gateway auto-detect, four-state progress chip, live hit stream, Add buttons per hit, 2,048-host large-subnet warning, subnet syntax cheat sheet at `/help/subnets.html`).
- **ws-scrcpy-web deployment mode** — General Settings gained a Managed / External toggle. Managed keeps today's auto-spawn behaviour; External lets Control Menu point at a Docker container or remote host via URL. Dependency management respects mode: Install/Update hidden in External mode.
- **`NetworkScanService` (singleton)** — holds a `ClientWebSocket` to ws-scrcpy-web's `/ws-scan` endpoint, fans scan events to every Blazor circuit spectating a scan. Two browser tabs attaching mid-scan share one scan session with snapshot replay.
- **`SubnetParser`** — CIDR / IP / range parser mirroring ws-scrcpy-web's behaviour byte-for-byte, error verbiage included.
- **Device SerialNumber auto-fill** — when a user adds a device from the Discovered panel, `AddFromDiscovery` now probes `ro.serialno` alongside the existing kind and model probes and pre-fills the Serial Number form field. Users who type a value during the probe window keep their entry (empty-check guard, same pattern as the existing Name auto-fill). Existing registered devices without a serial stay unchanged — backfill is intentionally out of scope.

### Changed
- `MainLayout.razor` page-title switch gained a specific case for `/android-power-tools` that sits above the generic `path.StartsWith("android")` fallback. Without this the breadcrumb rendered "Android Devices" for the new route because the prefix match order is load-bearing.
- **Scan Network** (Settings > Devices) now queries mDNS first via `adb mdns services`, enriches each hit with MAC from the ARP table (pinging any mDNS IPs not yet in ARP to force resolution), and falls back to ARP-only refresh for devices that don't advertise over mDNS. When a device is matched via mDNS, Control Menu updates both its IP address AND its stored ADB port — previously the port would remain stale when a device's wireless debugging port changed (common on Android 11+ phones that rotate ports per session). Discovered unregistered devices are surfaced in a new panel below the registered-devices table (see Added). Toast reports `N of M registered device(s) found`, `K ADB port(s) updated from mDNS`, and `X new device(s) on network` as applicable.
- **Adding a new device auto-removes it from the Discovered panel.** Without this, the user would see the device they just added still listed as "available to add" until the next scan.
- `ScrcpyMirror.razor` now uses ws-scrcpy-web's `/embed.html?device=<udid>` wrapper URL (shipped 2026-04-17 in ws-scrcpy-web 1.0.0); the legacy `#!action=stream&udid=...&embed=true` hash routing has been removed upstream. Both `Inline=true` (inline iframe) and `Inline=false` (popup window) share the same URL — `/embed.html` is always in embed mode.
- `docs/TECHNICAL_GUIDE.md` — Stream URL section updated; Android Devices module description updated to reflect mDNS-first discovery; AdbService method table loses `GetUsbDevicesAsync` and `ResetTcpPortAsync`, gains `ScanMdnsAsync`; dashboard descriptions note the `DeviceKind` hint being passed to `ScrcpyMirror`.
- `docs/manual-test-checklist.md` — Section 5 (Settings > Devices) rewritten into subsections covering the registered-devices CRUD, Scan Network behavior, Discovered panel flow, and delete→rediscover name-memory path. Dashboard sections 10 (Google TV) and 11 (Android Phone) now include DeviceKind-toolbar-default assertions.
- `README.md` License section updated from "Private project" to GPL-3.0-only with links to LICENSE and CONTRIBUTING; added a Security section pointing at SECURITY.md.
- **`Settings › Devices`** — `Scan Network` button split into `⟳ Quick Refresh` (today's silent mDNS+ARP, unchanged) and `📡 Scan Network…` (new modal). "Discovered on Network" section now mirrors the last scan: Full Scan replaces the list; Quick Refresh merges.
- **`WsScrcpyService`** — `StartAsync` short-circuits process spawn in External mode. `BaseUrl` resolves at start time based on mode.
- **`DependencyManagerService.CheckHealthAsync`** — ws-scrcpy-web entry pings `{url}/` in External mode instead of checking process state.
- **Scan UX redesign** — `Scan Network…` modal shrinks to a subnet picker + Start button. Chip, live hit stream, Cancel button, and per-row Add buttons now live on the `Settings › Devices` page between the registered-devices table and the Discovered section. Each Discovered row gains a `×` Dismiss button to clear unwanted entries without a fresh scan. Opening the modal during an active scan disables Start (label: "Scan in Progress…") but allows editing the subnet list for the next run.
- **Chip palette theme-aware** — `ScanProgressChip` swaps hardcoded hex for `color-mix` over `--info-color` / `--warning-color` / `--success-color` / `--danger-color` tokens so both dark and light themes render legibly now that the chip lives on the page background.
- **`.source-badge` token-driven** — follow-up to the chip palette work; the amber badge now uses `color-mix` over `--warning-color` instead of hardcoded `rgba(245, 159, 0, 0.15)` + `#d97706`.
- **`ScanMergeHelper.AddressKey(ip, port)`** centralizes the `"ip:port"` format used across the dismiss set, registered-IP exclude set, `_discovered` dedupe, adb-merge service name, and dismiss-click write. Previously each call site built the string inline, aligning by coincidence of `ws-scan`'s emission format. Future format drift (IPv6-literal brackets, zone suffixes) can be contained in the helper.
- **Shared ARP snapshot on `scan.complete`** — `FinalizeScanAsync` orchestrates enrichment in one ARP pass: `DetermineAdbMergeCandidatesAsync` (runs `adb devices`, filters), `BuildArpMapWithPingsAsync` (builds ARP, pings any missing IPs across both enrichment branches, rebuilds once), then `EnrichDiscoveredMacs` + `AppendAdbMergeRows` consume the shared map synchronously. Worst-case ARP shells drop from 4 to 2.
- **Scanner extraction** — The 120-line scan event dispatch + finalize orchestration that lived in `DeviceManagement.razor` moves into a new scoped service `ScanLifecycleHandler`, unit-testable without bUnit. The ~40-line Discovered panel markup moves into `DiscoveredPanel.razor`. Page responsibilities shrink to device CRUD, Quick Refresh, and modal wiring; handler owns Discovered / DismissedAddresses / StashedNamesByMac / Phase / LastProgress. Closes reviewer item I-1 (the `_discovered`-across-awaits race) — `AppendAdbMergeRows` now re-filters dismissed addresses at apply time, covering the window where a user dismisses a row during the ARP/ping finalize await.
- **Discovered panel inline Add** — rows now contain editable fields (Name, Type, Port, MAC, Serial, PIN) populated by on-mount ADB probes. Clicking Add saves the row directly — the per-device modal dialog is gone for this path (manual Add Device and Edit still use the modal). Each row reads `device-name-<mac>` from config directly, so a user's stashed name (from a prior delete) correctly restores on rediscovery regardless of scan timing. The former "Service" column is now "Source" and renders only a small `mdns`/`tcp`/`adb` badge — the raw mDNS service-name string is gone. Add button is theme-green, × button is theme-red (new `.btn-success` CSS rule mirrors the existing `.btn-danger` pattern using `var(--success-color)`).
- **Settings › Devices table alignment** — Device Management and Discovered tables now share a scoped `.data-table-fixed { table-layout: fixed; }` class so columns obey declared widths instead of auto-layout's content-based stretching. A 110px filler column on Discovered matches Managed's "Last Seen" slot so both tables total 1105px and widths aren't proportionally rescaled to fit the container. Editable fields inside fixed tables get `margin-left: -13px; width: calc(100% + 13px);` so input text aligns pixel-for-pixel with static-text cells in the column above — shifting the box (not the padding) keeps the input's internal border/padding intact while the text lands at cell-edge + 12px like everywhere else. Discovered header labels "MAC" / "IP" renamed to "MAC Address" / "IP Address" to match Managed. Final column widths (identical both tables): Status/Source 60, Name 130, Type 130, MAC Address 150, IP Address 110, ADB Port 70, Serial 150, PIN 65, Last Seen/filler 110, Actions 130.
- **Sidebar logo** — the "Control Menu" pill button in the sidebar header is now horizontally centered in the header instead of left-aligned with the nav items below. The 1px outline around the pill is transparent; the pill's subtle background-tint differential against the sidebar is enough visual separation. The collapse toggle (`<`) moves to absolute positioning on the right edge so the pill centers independently of it.

### Removed
- **`build.log`** — stale build output containing old `tools-menu` paths; added `build.log` to `.gitignore` to prevent future tracking.
- **`IScanLifecycleHandler.StashedNamesByMac`** and its backing implementation. The cache was populated by `PopulateStashedNamesAsync()` during `FinalizeScanAsync` (after `ScanCompleteEvent`), but `DiscoveredPanelRow` mounts during the scan on `ScanHitEvent` — the cache was always empty at row-mount time, so the fallback lookup in `ResolveInitialName()` never fired. Stashed-name restoration is handled correctly by `DiscoveredPanelRow.RunProbesAsync`, which reads `device-name-<mac>` directly via `IConfigurationService`. Handler ctor loses the `IConfigurationService` parameter (no longer needed). `DiscoveredPanel` / `DiscoveredPanelRow` lose the `StashedNamesByMac` parameter; `DeviceManagement.razor` drops the pass-through attribute. This also obsoletes two previously-planned follow-up items (B3 enumeration hardening and B4 parallelization of the stashed-name populate path) — the code they were going to harden no longer exists.
- `UsbSetupWizard.razor` and its stylesheet. The component existed to wire a freshly-flashed Android phone to wireless ADB over a USB cable (`adb tcpip <port>` → disconnect USB → connect over WiFi). ws-scrcpy-web's mDNS-based discovery replaces this flow end-to-end — phones advertise their ADB port over mDNS once wireless debugging is enabled once by the user, so the USB handshake dance is no longer necessary. The corresponding "Enable Wireless ADB" action-row on `PixelDashboard.razor` and the new-phone auto-trigger on `DeviceManagement.razor`'s Save flow have been removed. `IAdbService.GetUsbDevicesAsync` and `IAdbService.ResetTcpPortAsync` were deleted along with their implementations — both existed only to support the wizard.

### Fixed
- **`fix(a5)`** Scrcpy settings modal probe was sending `action=PROBE_DEVICE` but ws-scrcpy-web expects `action=probe` — probes silently failed with WebSocket close code 4002 ("Unsupported request"), leaving the DB empty and all modals stuck on hardcoded h264/raw fallback defaults. Fixed the action value; probe data now saves correctly to the database on first device dashboard visit.
- **`fix(a5)`** Video/audio codec dropdowns were hardcoded to all codecs (`h264/h265/av1` and `opus/aac/flac/raw`) instead of being filtered to what the device actually supports. Added `DetectVideoCodecs`, `DetectAudioCodecs`, and `EncoderMatchesCodec` methods using dot-delimited segment matching (ported from ws-scrcpy-web's `ConfigureScrcpy.detectVideoCodecs`/`detectAudioCodecs`/`encoderMatchesCodec`), handling both `c2.*` and `OMX.*` encoder naming conventions. Dropdowns now rebuild from probe cache on every modal open and after every re-probe.
- **`fix(a5)`** Encoder dropdown showed only "(auto)" with no intelligence. Added hardware-preferred encoder auto-selection (`PickBestEncoder`) matching ws-scrcpy-web's vendor-prefix logic (`.mtk.`, `.qcom.`, `.exynos.`, `.intel.`, `.nvidia.` = hardware). Encoder auto-selects on codec change and on modal open when no explicit encoder is saved.
- **`fix(a5)`** Audio source default was `playback` (device + stream); changed to `output` (stream only, silences device) as the correct default for most use cases.
- **`fix(a5)`** Added `sdkInt` field to `ScrcpyProbeResult` (from ws-scrcpy-web's updated probe response). Modal now gates audio UI by SDK version: audio checkbox disabled with tooltip "Android 11+ required" when SDK < 30; `playback` audio source hidden when SDK < 33 (no `--audio-dup` support).
- **`fix(b5)`** Cross-circuit sidebar nav desync: device mutations in one browser tab now update all open tabs' sidebar navigation in real-time. Replaced the scoped `IDeviceService.DevicesChanged` event with a singleton `IDeviceChangeNotifier` pub/sub service so `DeviceTypeCache` and `DeviceTypePresenceWatcher` receive notifications across Blazor Server circuits.
- **`fix(home)`** Home page module-card sub-link pills now render SVG-path icons as `<img>` tags instead of as literal path text. `Home.razor` had a two-branch icon renderer (Bootstrap `bi-*` or emoji) that predated the 5-SVG sidebar-icon switch in A1+A4; SVG paths like `/images/devices/device-list.svg` fell through the emoji branch and rendered as a `<span class="pill-emoji">` showing the raw path. Added a middle branch matching `Sidebar.razor`'s three-way logic (`bi-` / `/` or `.svg` / else) plus a new `.pill-icon-img` CSS class sized to 1em square with `object-fit: contain` and `flex-shrink: 0` so it scales with the pill's font-size context and doesn't collapse in narrow cards.
- **`fix(tablet,watch)`** `TabletDashboard.razor` and `WatchDashboard.razor` were cloned from `PixelDashboard.razor` during A1+A4 but the sibling `.razor.css` files were not cloned. Blazor scoped CSS attaches via a per-component `b-xxxxx` attribute, so none of Pixel's scoped styles (quick-actions card layout, status badges, mirror sizing) applied to the clones — the tablet dashboard rendered as a flat unstyled list with a tiny ~184x240 mirror. Copied `PixelDashboard.razor.css` to `TabletDashboard.razor.css` and `WatchDashboard.razor.css`. Watch still unverified on real hardware.
- **`fix(wizard)`** Setup wizard container `max-width: 960px → 1260px`. The Devices step's `DiscoveredPanel` table (10 columns summing to 1105px) was producing a horizontal scrollbar inside `.wizard-content` because the container was too narrow. Other 6 wizard steps (form-based) are unaffected since their internal content flows naturally at the wider width. `overflow-x: auto` stays as defense-in-depth.
- **`fix(scanner)`** `DiscoveredPanel` ADB Port column `width: 70px → 105px`. At 70px minus cell padding minus `<input type="number">` spinner arrows, the usable text area (~30px) fit "55" but clipped 4-digit ports like `5555`/`32851`. 105px gives comfortable room for 5-digit ports with breathing room; total column budget still sums under the available wizard-content width.
- **`fix(tables)`** `.data-table .actions` used `display: flex` on the `<td>` which overrode the cell's native `display: table-cell`, breaking row border continuity — the `border-bottom` traced the flex container's bounding box instead of flowing across the cell, producing a visible "cut-in" notch at the bottom-right of every row. Replaced with `text-align: right; white-space: nowrap;` plus `.data-table .actions > * + * { margin-left: 6px; }` (adjacent-sibling spacing replicates flex `gap: 6px` for inline-block buttons). Fix propagates to all three uses: `DiscoveredPanelRow`, `DeviceManagement`, `DependencyManagement`.
